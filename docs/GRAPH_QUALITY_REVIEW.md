# AnshuDoc 图谱质量与产品风险评审

本文记录当前 AnshuDoc MVP 在知识图谱、语义检索、PPT 大纲生成和产品定位上的主要问题，以及建议的演进方案。目标不是否定 MVP，而是明确哪些能力只是“可用原型”，哪些能力需要成为后续核心工程投入。

## 0. 产品方向：服务中小企业业务 AI 落地

AnshuDoc 不再以 InfraNodus 复刻、学术研究工具、学生学习工具或泛个人知识管理工具为参照。它属于 **Anshusoft Grove** 系列桌面产品的一部分，面向中小企业，作为业务 AI 落地服务的切入工具。

Anshusoft Grove 的长期方向是形成一套面向企业客户的 AI 落地方案，而不是单点工具集合。AnshuDoc 在这套方案中的角色是：

- 连接企业已有文档、制度、流程、项目资料和知识资产。
- 将非结构化文档转换为可检索、可追溯、可汇报的业务知识底座。
- 帮助企业从“有大量文档但难以复用”过渡到“能基于证据生成摘要、分析和汇报”。
- 为后续业务 AI 应用提供本地化、项目化、可解释的上下文来源。

### 从单点工具到企业知识网络

单个 AnshuDoc 的价值是让一个部门、一个项目或一个文件夹里的文档变得可理解、可检索、可汇报。但如果企业中不同部门都在使用 AnshuDoc，价值会从“个人/部门效率工具”升级为“企业知识网络”。

这种联通的特殊意义在于：

- **部门知识互通**：质量、研发、生产、销售、交付等部门各自沉淀的文档图谱，可以在授权范围内形成跨部门主题关联。
- **组织记忆形成**：企业过去的项目资料、客户问题、质量案例、交付经验不再只是文件，而是可召回、可解释、可复用的组织知识。
- **业务断点暴露**：跨部门图谱可以发现同一概念在不同部门中的定义差异、信息缺口、流程断点和重复建设。
- **经验复用**：新项目启动时，可以从历史项目、相似客户、相似问题中召回案例、风险点、方案和文档证据。
- **管理视角增强**：管理层可以从部门级图谱上升到公司级主题图谱，看到知识资产、业务风险、协作瓶颈和能力分布。

因此，AnshuDoc 不应只被设计成“本地文档分析器”。它的长期形态应支持：

- 本地优先的部门/项目知识库。
- 企业内部可控联通的知识索引。
- 基于权限的跨项目、跨部门语义检索。
- 从局部图谱汇聚到组织级知识地图。
- 为后续企业 AI Agent 提供可信上下文。

这个方向也解释了 Anshusoft Grove 后续产品线的意义：不同桌面工具不是孤立应用，而是从企业不同业务入口采集、整理和激活知识。

例如后续的 **AnshuMail** 可以成为邮件和沟通知识入口：

- 将客户邮件、内部讨论、需求变更、问题反馈、会议纪要等转为可追溯知识。
- 识别邮件中的客户、项目、问题、承诺、风险和后续动作。
- 将邮件上下文与 AnshuDoc 的项目文档图谱打通。
- 让“文档中的制度/方案”和“邮件中的真实业务过程”能够互相印证。

长期看，AnshuDoc 负责“静态文档知识”，AnshuMail 负责“动态沟通知识”。两者联通后，Anshusoft Grove 可以逐步形成面向中小企业的业务 AI 操作系统雏形：不是先做大而全平台，而是从企业每天真实产生的文档、邮件、项目资料和沟通记录中，持续构建可用的企业知识底座。

### 目标用户

明确服务企业用户，尤其是中小企业中的以下角色：

- 企业负责人、业务负责人、部门主管。
- 质量、生产、研发、项目、售前、交付、运营等业务团队。
- 内部知识管理、数字化转型、AI 落地负责人。
- 帮助企业做 AI 落地的顾问或实施团队。

明确不作为优先用户：

- 学校、学生、科研人员。
- 泛笔记用户。
- 以论文/文献研究为核心的学术场景。

### 核心业务价值

AnshuDoc 需要围绕企业场景定义价值，而不是围绕“图谱很酷”定义价值：

- **资料盘点**：快速理解一个项目、部门或业务流程下有哪些关键文档和主题。
- **知识复用**：把散落在 PDF、Word、Excel、Markdown 中的知识转成可检索上下文。
- **证据型摘要**：生成摘要、大纲和建议时能回到原文证据。
- **管理汇报**：把项目资料整理成管理层可读的 brief、汇报大纲、风险/机会分析。
- **AI 落地前置工作**：帮助企业先整理知识资产，再接入问答、流程助手、培训助手、质量助手等后续 AI 应用。

### 产品边界

AnshuDoc 当前阶段应定位为：

> 面向中小企业的本地项目文档智能工作台：把企业项目文件夹中的资料转为可探索的概念网络，并基于语义检索生成证据型摘要、节点解释和汇报大纲。

它不是：

- 学术知识图谱工具。
- 学生学习助手。
- 通用笔记软件。
- 直接替代企业知识库或 OA 的系统。
- 一键生成完美 PPT 的工具。

它是 Anshusoft Grove 企业 AI 落地方案中的“文档知识底座”和“项目知识整理入口”。

## 1. 概念提取质量存疑

### 问题

当前图谱构建仍以本地规则和简单共现为基础：

- 英文主要依赖正则 tokenization 和 stop words。
- 中文处理非常粗糙，连续汉字片段容易被当成概念。
- 关系主要来自固定窗口共现，而不是真实语义关系。
- LLM 概念提取依赖 prompt 和模型质量，不同模型结果差异明显。

这会导致：

- 中文文档里出现大量泛词、短语碎片或非概念节点。
- 高频词容易被误认为核心概念。
- 概念之间的边只能表示“邻近出现”，不能代表真实关系。
- 同一项目用不同模型构建，图谱稳定性可能不足。

### 解决方案

短期：

- 将本地规则定位为 fallback，不作为高质量图谱的主路径。
- 对中文引入更可靠的分词和短语抽取策略，例如 `jieba-rs`、`tantivy` tokenizer 或基于关键词抽取的 TextRank/YAKE 方案。
- 加强概念过滤：过滤泛词、动词化短语、过短概念、文档结构噪声。
- 对 LLM 提取结果增加规范化层：别名合并、大小写归一、同义词合并、低置信节点过滤。

中期：

- 将概念提取拆成两阶段：
  1. 候选概念召回：规则、统计、embedding、标题结构。
  2. 概念确认和归类：LLM 或小模型判断是否为有效概念。
- 为每个节点保存来源证据，包括文档、chunk、原句、抽取方式和置信度。
- 支持项目级词表和黑名单，让用户能修正概念质量。

长期：

- 构建可评估的 benchmark：用若干真实项目文档人工标注核心概念，持续比较不同抽取策略。
- 引入实体链接和领域词典，让“概念图谱”逐步升级为“领域知识图谱”。

## 2. 知识图谱深度不足

### 问题

当前图谱主要包含：

- 节点：概念、词、短语。
- 边：共现关系。

这更接近“概念网络”或“语义索引图”，还不是严格意义上的知识图谱。主要缺口包括：

- 缺少关系类型，例如“属于”、“影响”、“依赖”、“导致”、“用于”、“组成”等。
- 缺少实体属性，例如时间、角色、来源、指标、定义。
- 缺少事实三元组和证据链。
- 社区检测偏基础，对复杂文档的主题簇解释能力有限。

### 解决方案

短期：

- 产品描述上避免过度承诺，明确当前是“概念网络 + 语义检索 + 文档摘要”的 MVP。
- 在 UI 中区分边类型：
  - `co_occurs_with`：共现边。
  - `semantic_related_to`：语义相关边。
  - `llm_relation`：LLM 抽取的关系边。
- 为边保存 `source`、`confidence`、`evidenceChunkIds`。

中期：

- 引入关系抽取任务，让 LLM 从高价值 chunk 中抽取轻量三元组：

```json
{
  "subject": "AI quality management",
  "predicate": "requires",
  "object": "traceability",
  "evidence": "source sentence",
  "confidence": 0.82
}
```

- 只对高相关 chunk 做关系抽取，避免成本失控。
- 将图谱分层：
  - 概念层：高召回，适合探索。
  - 证据层：chunk 和引用。
  - 关系层：低召回高精度的实体关系。

长期：

- 支持用户确认/编辑关系，形成可维护的项目知识库。
- 对社区检测引入更成熟算法或后端预计算，如 Leiden/Louvain，前端只做展示。

## 3. 企业应用场景需要收敛

### 问题

当前 AnshuDoc 的能力覆盖面较宽：

- 文档库
- 知识图谱
- 语义检索
- 摘要
- 节点摘要
- PPT 大纲

但目标用户和核心工作流需要进一步聚焦到中小企业。企业客户不是为了“看图谱”而使用工具，而是为了完成具体业务任务：

- 快速理解一个项目文件夹里的资料结构。
- 从制度、规范、项目材料中提取可复用知识。
- 生成可追溯的管理摘要和汇报大纲。
- 支持业务 AI 助手、质量助手、项目助手等后续落地场景。

如果定位不清，容易变成“功能很多但每个都不够尖”的工具。

### 解决方案

建议优先收敛到一个 MVP 场景：

> 面向中小企业的本地项目文档智能工作台：把一个项目文件夹中的企业资料转换为可探索的概念网络，并基于语义检索生成证据型摘要和管理汇报大纲。

优先用户：

- 企业负责人或部门负责人。
- 项目经理、质量经理、研发负责人、交付负责人。
- 企业内部 AI 落地负责人。
- 为企业提供 AI 落地服务的实施顾问。

核心痛点：

- 企业资料分散在多个格式和文件夹中，难以快速理解。
- 新项目启动、客户交付、质量评审、内部培训前，需要快速整理资料。
- 普通 RAG 问答缺少结构化视图，管理层不容易看到主题、风险和证据。
- 从文档到管理汇报需要大量人工阅读、摘录和组织。

与通用笔记/知识库工具的差异：

- 本地优先，适合私有资料。
- 面向“企业项目文件夹”，而不是个人笔记系统。
- 同时提供图谱结构、语义证据和汇报输出。
- 可接 Ollama，本地模型可用。
- 后续可以纳入 Anshusoft Grove 的企业 AI 落地服务包。

汇报大纲功能的定位应调整为：

- 不承诺直接生成高质量 PPT。
- 先生成“基于证据的汇报大纲”。
- 后续再对接 PPT 生成服务或 MCP，成为企业汇报自动化链路的一环。

## 4. 性能和扩展性问题

### 问题

当前 MVP 为了保证前端可用，设置了节点和边上限：

```ts
const MAX_GRAPH_NODES = 450;
const MAX_GRAPH_EDGES = 1400;
```

这解决了页面卡顿，但也带来明显问题：

- 稍大的文档集会丢失大量低频但重要的信息。
- 前端力导向布局在节点多时成本很高。
- 2D/3D 图谱展示和真实知识库规模之间存在矛盾。
- 当前向量检索是普通 SQLite 表 + Rust 暴力余弦相似度，没有索引，数据变大后会慢。

### 解决方案

短期：

- 保留全量数据在 SQLite 中，前端只展示裁剪后的子图。
- 默认展示：
  - 与当前问题相关的子图。
  - 当前社区子图。
  - Top N 重要节点。
- 明确 UI 文案：当前图谱视图是“聚焦视图”，不是全量图谱。

中期：

- 引入图谱查询 API：
  - 按节点展开一跳/两跳。
  - 按社区查看。
  - 按问题召回子图。
  - 按文档过滤。
- 2D 视图继续使用高性能渲染库，但布局预计算应放到后端或 worker。
- embedding 检索从暴力计算迁移到 sqlite-vec 或其他 HNSW/ANN 索引。

长期：

- 将存储层拆成：
  - SQLite：元数据、文档、chunk、摘要、图谱边。
  - sqlite-vec：向量索引。
  - 可选图数据库：只在确实需要复杂图查询时引入。
- 支持大项目的渐进加载和虚拟化展示。

## 5. 推荐演进路线

### P0：稳定 MVP 定位

- 明确产品描述为“面向中小企业的本地项目文档智能工作台”。
- 不把当前共现图谱描述成完整知识图谱。
- 完成 Markdown 大纲模式，减少 JSON schema 失败带来的 fallback。

### P1：提高摘要和大纲质量

- 构建时生成 chunk embedding。
- 项目摘要、节点摘要、关联摘要、PPT 大纲都使用语义召回证据。
- 大纲生成优先 Markdown，而不是强制 JSON。
- 禁止模型使用证据外信息。

### P2：提高图谱质量

- 中文分词和短语抽取升级。
- LLM 概念抽取结果加入置信度、来源证据和去噪。
- 节点支持用户编辑、合并、隐藏。

### P3：引入真实关系抽取

- 在高相关 chunk 上抽取轻量三元组。
- 边类型从单一共现扩展为 typed relations。
- 建立证据链，让每条关系都能回到原文。

### P4：扩展性能

- sqlite-vec 接入向量索引。
- 子图查询和渐进加载。
- 图谱布局预计算或 worker 化。

## 6. 结论

当前 critique 是成立的。AnshuDoc 现在更像“概念网络 + 语义检索 + 企业项目文档工作台”的 MVP，而不是成熟知识图谱系统。

但这并不意味着方向错误。合理的路线是：

- 不夸大共现图谱。
- 用 embedding 提升证据召回。
- 用 LLM 做证据约束下的摘要和大纲。
- 逐步从概念网络演进到带关系类型和证据链的知识图谱。

只要产品定位收敛到中小企业业务 AI 落地，图谱质量、证据召回和企业工作流持续增强，这个方向仍然有明确价值。

AnshuDoc 的后续目标不是成为一个孤立工具，而是成为 Anshusoft Grove 系列产品中的第一个业务入口：先帮助企业整理项目文档和知识资产，再逐步扩展到企业内部问答、流程分析、培训材料生成、质量管理辅助、项目汇报和业务 AI 助手等完整落地方案。
